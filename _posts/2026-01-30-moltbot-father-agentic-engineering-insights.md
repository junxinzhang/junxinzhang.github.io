**"大公司要用好AI，需要先来一次大重构！"**

**"超级个性化助手，是反OKR的！"**

13年创业，3年"燃尽"，去年5月开始萌生造 Clawdbot（现已更名 Moltbot）的想法。一开始担心大厂会很快做出来，结果大厂却迟迟没动作，最后自己忍不住下场了。

近期伴随着 Clawdbot 的爆红，创建者 Peter Steinberger 忙到严重睡眠不足，一边忙着给它换个新名字 Moltbot，一边接受了不少播客的邀请。

昨天晚上，知名技术播客《The Pragmatic Engineer》跟 Peter 来了一场 114 分钟的对谈。

![Agentic Engineering 概念图](/assets/images/screenshot-20260130-peter_steinberger_codex.webp)

*Peter 的工作方式：同时调度多个 AI Agent，在心理上比单纯写代码更累*

而开头提到的"Clawdbot诞生的故事"，相信看过上篇[《Clawdbot刷屏AI圈，我为什么劝你别急着用》]({{ site.baseurl }}/clawdbot-why-you-should-wait)的朋友已经了解了。

**播客下半场，才是这场对谈的重点。可以说，所有反行业共识的东西全在这里了。**

---

## 一、不要叫我"Vibe Coding"，请叫"Agentic Engineering"

首先，作为一个现象级 AI 产品的 solo 开发者，Peter 非常不喜欢"Vibe Coding"这个词。

> **"我做的是 Agentic Engineering，带个小星号。"**

他解释说，真正的 Vibe Coding 是那种凌晨三点开始、完全放飞自我的状态。而他的工作方式完全不同：

**他同时调度 5 到 10 个 Agent。**

每个 Agent 都在干不同的事情。他的大脑不停切换，从这个子系统跳到那个功能，又跳到另一个地方。

> "这其实更累。因为我不再是管一个人，而是同时在管五个、十个。"

通常的工作节奏是这样的：

1. 设计一个新子系统或新功能
2. 知道 Codex 大概需要 40 分钟到 1 小时才能跑出来
3. 把方案想清楚，交给它去"煮"
4. 转身去干别的任务
5. 过一会儿回来看结果

**他形容这种感觉像"星际争霸"——有主基地，也有分基地，不断给你输送资源。**

也像国际象棋大师同时下二十盘棋，走到一盘前，看一眼局面，做个决策，遇到强对手就多停一会儿。

---

## 二、"架构师"这个词也不对

Peter 同样不太喜欢"程序员在 AI Coding 时代的新定位"的称呼："架构师"。

> **"我不太用'架构师'这个词，我更喜欢'builder'。"**

他发现用 AI 特别顺的人和特别痛苦的人，差异很明显：

| 类型 | 特征 | AI 使用感受 |
|-----|------|-----------|
| **Builder 型** | 更在意结果和产品体验，底层管道关心结构但不纠结细节 | 进展顺利 |
| **纯技术型** | 特别热爱写难算法，喜欢解决纯技术难题，不太喜欢做完整产品 | 最抗拒 AI，也最容易沮丧 |

原因很直接：**AI 恰恰擅长解决那些"难问题"。**

> "这一年里，我学到的系统架构和设计知识，比过去五年加起来都多。这些模型里塞进了巨量知识，几乎任何问题都能问到答案，前提是你得知道该问什么。"

---

## 三、瀑布式开发的"精致混乱"

这里是 Peter 最毫不留情面的批评之一：

> **"我看到很多人会先搭一整套复杂的编排层：什么自动建工单、Agent 处理工单、Agent 再给另一个 Agent 发邮件，最后堆出一坨精致的混乱。图什么？"**

他痛斥 AI 圈内非常常见的"编排、拆解"做法：

> "他们可能花几个小时设计 spec，然后机器用一天把东西'造'出来。**我不信这套能行。这本质上还是瀑布式软件开发，我们早就知道它行不通。**"

那他是怎么做的？

> "我通常从一个想法开始，甚至会刻意'欠提示' Agent，让它做点出乎意料的事，给我新想法。可能 80% 的结果都不怎么样，但总会有一两个点让我意识到'这个角度我没想过'。然后我不断迭代、塑形。"

**他把这个过程比作雕塑：**

> "从一块石头开始，不断凿，慢慢一尊雕像从大理石里浮现出来。这就是我对'构建'的理解。"

---

## 四、高效使用 Agent 的关键：建好验证闭环

这是整场对话中最有价值的技术洞察之一。

![AI Coding 验证闭环](/assets/images/screenshot-20260130-verification_loop_ai.webp)

*高效使用 Coding Agent 的关键：让 AI 能自己测试、自己 debug*

> **"高效使用 coding agent 的关键，其实就一句话：你得把反馈闭环建好。它必须能自己测试、自己 debug，这是最大的秘密。"**

这也解释了一个很多人困惑的问题：

**为什么 AI 在写代码上表现很好，但在写作上常常平平？**

因为代码可以被验证：可以编译、跑测试、检查输出。而创意写作很难验证。

Peter 的实际做法：

> "比如昨天我在 debug 一个问题：Mac App 找不到远端 gateway，但同样的 TypeScript 代码却可以。Mac 应用调试特别烦。"
>
> "现在我就直接说：你给我建一个 CLI，专门用来 debug，走完全一样的代码路径。然后它就开始跑，一个小时后搞定，告诉我这里有 race condition，那里有配置问题。"

**关上闭环的意义：**

让 Agent 能验证自己做的事情。只要你把验证机制搭好，就能形成一个非常完美的反馈回路。

> "哪怕是现在，我在做网站时，也会把核心逻辑设计成可以通过 CLI 跑起来，因为浏览器里的反馈循环实在太慢了，你需要一个能高速迭代的 loop。"

## 五、从 Claude Code 到 Codex——为什么切换？

Peter 透露了他的工具选择变化：

> **"一开始是 Claude Code，然后我彻底上瘾了。但现在在复杂应用里写代码，Codex 明显更好。"**

两者的核心差异：

| 对比维度 | Claude Code | Codex (OpenAI) |
|---------|-------------|----------------|
| **上下文理解** | 可能读三份文件就自信开始写代码 | 会安静地读十分钟文件 |
| **首次成功率** | 第一次跑往往不通，需要来回修补 | 绝大多数时候一次就对 |
| **工作节奏** | 快速反馈，更"互动" | 更慢，但结果更稳 |
| **适用场景** | 通用计算、快速原型 | 复杂应用、大型代码库 |

> "用 Claude Code 的时候，你得用一种不太一样的方式工作。它确实更快，但生成的东西往往第一次跑不通，可能忘了同步改另外三处，要么直接 crash。"

**他给的建议：**

> "如果你只用一个终端，Codex 的等待体验确实让人受不了，但我更愿意要这种方式——因为我可以同时跑多个。"

## 六、大公司要用好 AI，需要先来一次大重构

这可能是整场对话中最反直觉的观点。

![企业 AI 转型](/assets/images/screenshot-20260130-enterprise_ai_restructure.webp)

*新世界需要的是那种有完整产品视角、什么都能干的人——而大公司往往不存在这样的角色*

> **"我觉得大公司会非常难以高效地采用 AI，因为这要求彻底重塑公司运作方式。"**

他举了一个直接的例子：

> "比如在 Google，你要么是工程师，要么是经理；想同时决定 UI 长什么样，这个角色不存在。"

**新世界需要的人才画像：**

- 有完整产品视角
- 什么都能干
- 数量要少得多
- 必须具备极高的自主性和能力

> "理论上，公司规模可以砍到原来的 30%。这很吓人，从经济角度看一定会引发混乱，很多人会很难在新世界里找到位置。"

**结论：**

> "所以我一点也不惊讶现在的大公司用不好 AI。他们确实在用，但要真正用好，得先做一次大重构——**不只是代码库，还有公司本身**。"

---

## 七、PR 必须附 Prompt——代码本身价值在缩水

这是另一个让人印象深刻的观点。

![Prompt is the new Code](/assets/images/screenshot-20260130-prompt_over_code.webp)

*AI 时代代码的 PR 本身价值在缩水，Prompt 反而才是更高信号量的东西*

> **"当我看到一个 PR 时，我更感兴趣的其实是 prompt，而不是代码本身。"**

Peter 现在要求所有提交代码的人一起提交 prompt：

> "有些人会照做，而我会花更多时间读 prompt，而不是读代码。"
>
> "对我来说，prompt 才是更高信号量的东西：你是怎么得到这个结果的？你具体问了什么？中间做了多少引导？这些比最终代码本身更能让我理解输出。"

**对提需求方式的改变：**

> "如果有人想要一个新功能，我会让他先写一个 prompt 需求，把它写清楚。因为我可以直接把这个 issue 丢给 agent，它就会帮我把功能做出来。"

---

## 八、好的 Prompt 不是预先设计的，是协作出来的

很多人以为 Peter 是"Prompt 高手"，一定有精心设计的 Prompt 模板。

**事实正好相反。**

> "这并不是提前精心预备好的，而是跟 Agent 之间的协作规划出来的。"

他的工作方式：

> "我会说，我们看看这个结构，有哪些选项？这个特性考虑过没有？因为每一轮 session，模型一开始对你的产品是零认知的，你得给它一些指路标志，让它去探索不同方向。"

**关于"planning mode"：**

> "你并不需要什么 plan mode，我就是一直聊天，直到我说'build'，它才会真的开始构建。有一些触发词，它们确实都挺'饥渴'的，但只要我说'我们讨论一下'或者'给我一些选项'，它们就不会动手。"

---

## 九、超级个性化助手——这是反 OKR 的

Peter 透露了他真正想做的东西：

> **"我想要一个'超个性化'的助手，不是那种早上给你发邮件、列三件待办的，而是真正理解我的。"**

他描述的场景：

- 你见了朋友，回家后它会问你"刚才那次见面感觉如何？"
- 提醒你"你三周没联系 Thomas 了，我注意到他现在在城里，要不要打个招呼？"
- 甚至问你"你每次见到某个人都会情绪低落，这是为什么？"

> **"非常私人的那种，几乎是反 OKR 的。像电影《Her》，而且我确信技术会走到那一步。"**

原因：

> "模型越能看到更大的上下文，就越能识别模式。即便它们只是矩阵计算，没有灵魂，体验上却常常不一样。"

---

## 十、为什么不喜欢 MCP？

在工具选择上，Peter 有一个反主流的立场：

> **"MCP 最大的贡献是逼着公司把 API 打开了，但这个概念本身挺蠢的。"**

他的核心批评：

1. **上下文污染**：你得在会话一开始就把所有工具的所有函数和说明全塞进上下文
2. **无法过滤**：模型先问"有哪些城市"，返回 500 个城市，但 MCP 机制下它没法过滤
3. **无法脚本化**：没法链式调用，没法写"一条命令拿到所有 25 度以上的城市，再筛选、再打包"

**他的替代方案：**

> "我希望一切都是 CLI。模型其实特别擅长用 bash。"
>
> "CLI 就不一样，它可以直接用 jq 过滤，只拿需要的。"

他甚至自己写了个小工具 `make-porter`，用 TypeScript 把 MCP 转成 CLI，直接打包用。

## 十一、Token 消耗的真相：本质是 TS 在搬 JSON

对于大家关心的成本问题，Peter 解释了技术本质：

> "说到底，这东西最后就是 **TypeScript 在搬 JSON**。"

他的系统架构：

> "我只是从 LLM 那里拿文本，存到磁盘，再把文本发到 WhatsApp——现在还有 Teams、Slack、Discord、Signal、iMessage。核心就是：我在不同形态之间搬运文本，可能走不同 provider，有不同 agent，有 agent loop，有很多配置，管道非常多，但里面没有哪一件事本身特别难。"

---

## 十二、孵化流程：bootstrap 文件的秘密

Peter 透露了如何让 Clawdbot 孵化出一个个性鲜明的私人 Agent：

> "我给模型加了一个 bootstrap 文件，告诉它：你现在'诞生'了，要建立身份和'灵魂'，把用户的价值观放进去。"

**孵化过程：**

1. 模型会说"你好"，会问"我是谁？我叫什么？"
2. 接着它会问：什么对你重要？你在做什么？
3. 完成引导后，它会删掉 bootstrap 文件
4. 生成 `user.md`、`soul.md`、`identity` 文档
5. 这些文档会随着互动不断演化、微调

> "那一刻他们不再觉得自己是在和 GPT-5.2 说话，而是在和自己的朋友聊天。"

**另一个魔法：**

> "你根本不需要手动改配置，因为 agent 能改自己的配置。你直接跟它说'更新你自己'，它会拉最新版本、完成更新，然后回来告诉你'我有新功能了'。"

---

## 十三、给应届毕业生的建议：保持无限的好奇心

面对"AI 会不会抢走新人的饭碗"这个问题，Peter 只给了一条建议：

> **"请保持无限的好奇心。"**

他的理由：

> "你现在拥有一台'无限耐心的机器'，它可以把一切给你解释清楚。你可以不停地问：为什么要这么设计？为什么是这个结构？通过这种方式建立系统层面的理解。"

**新人的优势：**

> "他们没有被过往经验'污染'。他们会用 agent 做出很多我们根本想不到的事情，因为他们并不知道'这原本是行不通的'。等他们这么用的时候，可能它已经真的行得通了。"

---

## 十四、传统上手文档，不再是优先事项

Peter 还提到了一个令人惊讶的变化：

> "我没有写传统的 onboarding 文档，而是用 Claude Code 的方式：agent 会拉代码、读文档、写配置、把一切都 setup 好，包括 launch agent，全程不需要人工步骤。"

原因：

> "这个产品本身就是 agent 构建的。它的结构、命名方式，完全符合 agent 的预期。这些模式在模型的权重里已经被'编码'过了，所以 agent 在这个仓库里行动得非常顺畅。"

**最终的 onboarding 变成了一句话：**

> "把这个 prompt 输入给你的 agent。"

## 核心观点总结

1. **Vibe Coding ≠ Agentic Engineering**：真正的工作方式是同时调度多个 Agent，在心理上比单纯写代码更累

2. **瀑布式 + AI = 精致的混乱**：先设计 spec 再让 AI 执行的方式本质上还是瀑布开发，行不通

3. **验证闭环是核心秘密**：AI 在代码上表现好是因为代码可验证，把反馈机制建好是关键

4. **Codex 已超越 Claude Code**：在复杂应用中，Codex 的深度理解能力和首次成功率更高

5. **大公司需要大重构**：不只是代码库，还包括组织架构——需要"什么都能干"的复合型人才

6. **Prompt 比代码更有价值**：要求 PR 必须附带 Prompt，因为那才是更高信号量的东西

7. **超级个性化助手是反 OKR 的**：真正理解你的 AI 助手需要访问非常私人的上下文

8. **CLI 优于 MCP**：MCP 污染上下文且无法脚本化，CLI 更适合 AI 使用

9. **新人需要好奇心**：有一台"无限耐心的机器"可以解释一切，关键是知道该问什么

---

## 写在最后

当一个经历过完整创业周期、真正带过团队、对工程质量极度敏感的人，开始把"写代码"这件事大规模交给 Agent，会发生什么化学反应和二阶变化？

Peter Steinberger 用 Moltbot 给出了一个答案：

**技术消失了。**

你只是跟手机里的一个朋友聊天，它无限资源，能访问你的邮箱、日历、文件，能帮你建网站、做行政、抓网页、给朋友打电话、给商家打电话。

**你不需要理解任何复杂性，上下文全都自然融合。**

这可能就是 AI 时代"原生产品"该有的样子。

而对于我们这些开发者、创业者来说，真正的问题不是"要不要用 AI"，而是：

**你愿不愿意让 AI 真正了解你？**

**你敢不敢把那些私人的、反 OKR 的上下文交给它？**

这个问题的答案，可能决定了你能把 AI 用到什么程度。
